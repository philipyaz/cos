# Cos — repo guide (read this first)

Cos — a personal "chief of staff": a writable **board** (`board/`, Next.js + a JSON store) for
work+life to-dos, a knowledge **vault** (`vault/`, the LLM-Wiki pattern), a prompt-injection
**guard** (`guard/`, FastAPI sidecar), semantic **search** (`search/`, sidecar), and **MCP**
servers (`mcp/`) that expose it all to agents. Feature docs live in the **MkDocs site** under
`docs/` (Material theme, `mkdocs.yml`, published to GitHub Pages): e.g. `docs/architecture/spec.md`,
`docs/features/board.md`, `docs/security/guard.md`, `docs/reference/search.md`,
`docs/architecture/hierarchy.md`, `docs/reference/migration.md`. `README.md` (the repo
landing/quickstart) stays at the root. Setup is driven by the skills in `.claude/skills/`
(`mcp-bridge-setup`, `guard-setup`, `backup-recovery`).

Every setup/skill shell step begins with `source "$(git rev-parse --show-toplevel)/config/load-config.sh"`,
which resolves `REPO_ROOT` from git and exports machine config from `config/cos.env` (generated by
`cos-setup`) — paths, ports, and URLs (`REPO_ROOT`, `BREW_PREFIX`, `VAULT_DIR`, `BOARD_URL`, the bridge/
sidecar ports + URLs, etc.) so nothing is hardcoded. The `config/` dir splits by concern: `cos.env`
(machine paths/ports, gitignored), `secrets.env` (the Anthropic API key), `settings.json` (board prefs),
and `auto-sync.json` (router switch); only the `*.example` files are committed.

## Documentation — write it in MkDocs, not loose Markdown

All project documentation lives in the **MkDocs site** (`docs/`, Material theme, config in
`mkdocs.yml`), published to GitHub Pages by `.github/workflows/docs.yml` on every push to `main`.
When you add or change docs:

- **Add/edit pages under `docs/`** and wire each new page into the `nav:` in `mkdocs.yml`. Do
  **not** create new top-level `*.md` files at the repo root for documentation — that is the
  scattered-Markdown pattern we deliberately moved away from. If you're about to write a
  `FOO.md` at the root, it belongs under `docs/<section>/foo.md` instead.
- **Keep only these at the root** (GitHub surfaces them; they are not site content): `README.md`,
  `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`, `CLAUDE.md`. Component `README.md`s (`mcp/*/`, `guard/`, `search/`, `backup/`, `tests/`) stay
  next to the code they document.
- **Cross-link with relative `.md` paths** (e.g. `../security/guard.md`) so `mkdocs build --strict`
  validates them; link to repo *source* files with absolute GitHub blob URLs.
- **Generated pages** (e.g. `docs/reference/labels.md` from `scripts/gen-labels-doc.mjs`) — edit
  the generator/source and re-run it; never hand-edit the generated page.
- **Preview / validate locally** (the repo uses `uv`): `uvx --with mkdocs-material mkdocs serve`
  to preview, `uvx --with mkdocs-material mkdocs build --strict` to fail on any broken link.
