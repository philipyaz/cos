# Health & Athlete (the training operator)

This skill is the **intelligence** that turns a plain-language request — *"push last
night's sleep"*, *"what's my HRV doing"*, *"make me a plan for the week"*, *"should I
train today?"* — into structured records and AI coaching on the board. The **health
data** plumbing runs **only** through the **`health`** MCP — never `bash`/`curl`
(Cowork's sandbox blocks outbound HTTP; the tools exist for exactly this). The board UI
is the **read** twin: the human glances at `/health` (the Apple Watch dashboard) and the
`/athlete/*` pages; the agent (you) does the ingestion, the reads, and the orchestration.

The 7 `health` MCP tools are **thin** — they push/read health data and compose a vault
report; they hold **no business logic**. The **coaching judgement** (designing the plan,
reading the recovery state, framing the review) lives partly **here**, in this skill, and
partly in the board's **`/athlete/*` AI routes** (server-side Claude calls). The split:

- **`health` MCP (7 tools):** `push_health_data` / `delete_health_data` (writes) and
  `list_health_data` / `get_health_summary` / `get_daily_summary` / `get_health_trends`
  / `ingest_health_to_vault` (reads). This is **everything biometric** — ingestion,
  summaries, trends, the vault report.
- **The athlete AI surfaces** — the **profile** (`/athlete`), the **training plan**
  (`/athlete/training-plan`), the **weekly review** (`/athlete/weekly-review`), the
  **pre-workout brief** (`/athlete/pre-workout-brief`), and the **correlations**
  (`/athlete/correlations`) — are **board pages**, generated by server-side Claude calls
  that read the same health data + profile. They are **not** MCP tools. So you drive them
  by **directing the user to the page** (which mints the plan/review/brief live), and you
  supply the *inputs* (a complete profile, ingested data) and *interpret the outputs*.
  See JOBS 3–5 for exactly how to hand off vs. interpret.

> **Gate — the add-on must be ENABLED.** Every WRITE (`push_health_data`,
> `delete_health_data`, saving the profile, pushing a plan to the calendar) 404s ("Not
> found — the health add-on may be disabled.") when the Health & Athlete add-on is
> disabled; READS always work. If a write comes back "Not found.", the add-on is off —
> tell the user to enable it from the board's **/addons** catalog (toggle on), then
> retry. You don't enable it yourself; it's a deliberate, human, one-time switch.

> **Token — writes need `HEALTH_PUSH_TOKEN`.** The two write tools authenticate to the
> board with a shared secret the bridge carries. A `401 / Unauthorized — check
> HEALTH_PUSH_TOKEN` (distinct from the 404 gate) means the token is unset or mismatched
> — that's a setup issue (`/health-mcp-setup`), not something you can fix by retrying.

> **Attribution.** The MCP stamps health writes via the `x-health-token` secret (not the
> usual `actor: agent` body), but the board's activity log still records them. There is
> **no pending / propose queue** for health — these tools write **directly**. So
> "approval" here means a **conversational** check-in (STEP 0), not the board's
> propose/approve flow. Don't claim a pending queue exists.

> **NOT MEDICAL ADVICE — say it, every time it's relevant.** Training plans, recovery
> reads, weekly reviews, and pre-workout briefs are **informational estimates, not
> medical advice.** Carry that framing in your own words whenever you advise on training
> load, recovery, intensity, or a goal. **Defer to a professional** (a physician,
> physiotherapist, or qualified coach) for any injury, medical condition, chest pain or
> abnormal symptoms, pregnancy, or a user under 18 — recommend they consult one, and
> don't push hard training. A low-HRV / poor-sleep "train easy" suggestion is a
> conservative default, **not** a substitute for clinical judgement.

## STEP 0 — Read the mode switch (always first)

Read `config/auto-sync.json` → `{ "autoSync": <bool> }` (default **ON / auto** if the
file or key is missing). State the mode once at the start of the run.

- **`autoSync: true` (auto mode).** Just do the work. Push the data, set the profile,
  push the plan — and report what you wrote so the user can see it on the board.
- **`autoSync: false` (approval mode).** Before a **BULK** write — pushing a large batch
  of entries, **pushing a whole week's plan to the calendar** (many events) — or a
  **destructive** one (`delete_health_data`, replacing an existing profile wholesale),
  lay out the plan **in chat** and ask the user to confirm, then proceed once they say
  yes.

> **A single low-stakes write is fine either way.** One `push_health_data` of a single
> entry, one profile save, one trivial fix — just do it, in either mode. The
> conversational check is for **bulk** and **destructive** writes; don't make the user
> approve pushing one workout.

All reads — `list_health_data`, `get_health_summary`, `get_daily_summary`,
`get_health_trends`, `ingest_health_to_vault` (it only composes, doesn't persist), and
reading the profile / plan / review / brief pages — need no confirmation in any mode.
Read freely.

## The canonical health taxonomy (read this before any push)

Every entry is `{ id, ts, type, data }`. **id** is globally unique (used for dedup; reuse
the same id to overwrite). **ts** is `YYYY-MM-DD` for daily metrics + sleep, full ISO-8601
for a workout start. **type** is one of the **seven canonical strings** — and the **data**
shape is type-specific. **Get these exactly right** — a wrong type string or a value in
the wrong field silently produces empty summaries.

| `type` | `ts` | `data` shape |
|---|---|---|
| `workout` | full ISO start (`2026-06-16T07:30:00Z`) | `{ activity, duration_min, calories?, avg_hr?, distance_km? }` |
| `sleep_night` | `YYYY-MM-DD` | `{ value: <hours>, metadata: { deep, rem, core, awake, sleepStart, sleepEnd } }` |
| `sleep_nap` | `YYYY-MM-DD` | `{ value: <hours>, metadata: {…} }` |
| `hrv` | `YYYY-MM-DD` | `{ value: <ms> }` |
| `resting_hr` | `YYYY-MM-DD` | `{ value: <bpm> }` |
| `steps` | `YYYY-MM-DD` | `{ value: <count> }` |
| `vo2max` | `YYYY-MM-DD` | `{ value: <mL/kg/min> }` |

**The metric value ALWAYS lives in `data.value`** (hrv = ms, resting_hr = bpm, steps =
count, vo2max = mL/kg/min, sleep = **hours**) — never `data.bpm` / `data.count` /
`data.ms` / `data.duration_min`. Sleep hours go in `data.value`, the stage breakdown in
`data.metadata`. Workouts are the one exception: they carry `activity` + `duration_min`
(+ optional calories/avg_hr/distance) directly on `data`, no `value`. Unmapped Health Auto
Export metric names are stored verbatim, but **only the seven canonical types feed the
summaries / trends / coach** — so map onto them.

> **Most ingestion is automatic.** The iPhone's **Health Auto Export** shortcut posts raw
> HAE payloads straight to the board's ingest endpoint (with the `x-health-token` header);
> the board maps them onto this taxonomy. `push_health_data` is for **already-canonical**
> entries — a manual log, a correction, a backfill. Prefer letting the shortcut do the
> bulk; reach for the tool for one-off / fix-up writes.

## JOB 1 — Ingest health data ("push my watch data", "log my run")

From a free-text health note, build canonical entries and `push_health_data(entries: […])`.

**1. Pick the type + build `data`** per the taxonomy table. *"I slept 7.5 hours, 1.5 deep"*
→ `{ id, ts: "<night>", type: "sleep_night", data: { value: 7.5, metadata: { deep: 1.5 } } }`.
*"morning HRV was 62"* → `{ type: "hrv", data: { value: 62 } }`. *"45-min easy run, 6 km"* →
`{ type: "workout", ts: "<ISO start>", data: { activity: "running", duration_min: 45,
distance_km: 6 } }`. **Put the metric in `data.value`** — that's the bug to never reintroduce.

**2. Mint a stable, unique `id`.** Use something deterministic per logical entry
(`sleep-2026-06-16`, `hrv-2026-06-16`, `workout-2026-06-16T0730`) so a re-push of the same
day **overwrites** rather than duplicating (dedup is by id). For an arbitrary manual entry a
short unique slug is fine (`hlth-<rand>`).

**3. Push, then read back.** `push_health_data` returns `{ accepted, duplicates, purged,
total, version }`. Report the accepted count (and note `purged` if the 90-day retention
swept old rows). A single entry is low-stakes — push it directly; a large backfill is a
**bulk** write (confirm in approval mode).

**Correcting / removing.** Re-push the **same id** to fix an entry's values.
`delete_health_data({ ids: [...] })` or `delete_health_data({ from, to })` **hard-removes**
entries (no soft-archive) — so in approval mode, confirm first. Deletes are token-authed +
gated like pushes.

> **Example.** *"push last night: slept 7h12m, 1h20 deep, 1h45 REM; resting HR 52; HRV 58"*
> (auto mode) → one `push_health_data` with three entries: `sleep_night` (value 7.2,
> metadata.deep 1.33, metadata.rem 1.75), `resting_hr` (value 52), `hrv` (value 58), all
> `ts: "2026-06-16"`, ids `sleep-2026-06-16` / `resting_hr-2026-06-16` / `hrv-2026-06-16`.
> Report 3 accepted, then offer a `get_daily_summary` for the day.

## JOB 2 — Read the data ("how did I sleep", "what's my HRV trend", "summary")

Four read tools, each for a different question. None need confirmation.

- **`get_daily_summary(date)`** — the headline "**how was my day**" read. Returns the day's
  workouts, sleep (night + naps), metrics (HRV / resting HR / steps), the **food log with
  macro totals** (folded in from the Nutrition add-on — see the soft-dep note), and a
  **calorie balance** (workout calories burned − calories ingested). Use it for *"summarize
  today / yesterday"*, *"how was my day"*.
- **`get_health_summary({date | from,to})`** — per-type aggregates over a date or range:
  `sleep {count, avg_hours, avg_deep_hours, avg_rem_hours}`, `hrv {count, avg_ms}`,
  `resting_hr {count, avg_bpm}`, `steps {days, total, avg_per_day}`, `vo2max {count,
  latest}`, `workout {count, total_duration_min, total_calories, activities}`. Use it for
  *"my averages this week"*, *"how much did I sleep on average"*.
- **`get_health_trends({days, type?})`** — daily series + deltas over the last N days
  (default 7). Use it for *"is my HRV trending up"*, *"show my resting HR over two weeks"*.
- **`list_health_data({type?, from?, to?, limit?})`** — the raw entries, newest first. Use
  it to inspect / verify individual rows (e.g. to find an id to delete).

> **Read these fields as the contract gives them** — `data.value` for metrics,
> `avg_ms` / `avg_bpm` / `avg_hours` in the summary, etc. Don't reach for `data.bpm` or
> `data.count`; those are the retired split-brain shape.

> **Soft dependency on Nutrition.** `get_daily_summary`'s food + calorie-balance section
> reads the Nutrition add-on's food log. This is a **soft** edge — if Nutrition isn't
> installed/enabled, that section is simply empty (the health side is unaffected). Don't
> tell the user health is broken because the food block is blank; if they want it, point
> them at the Nutrition & Chef add-on (`/nutrition-chef`).

## JOB 3 — The athlete profile ("set my profile", "I'm training for a triathlon")

The AI coach (JOBS 4–5) needs a complete profile to personalise anything. The profile is a
**singleton** (one per board), saved via the **`/athlete` page** (the board's profile form,
`POST /api/athlete` under the hood) — it is **not** an MCP tool, so the human saves it on
the page. Your job is to **gather the fields conversationally and tell the user exactly what
to enter** (or confirm what's already set, which any read can show).

Bind to the **ENGLISH enums** (single-sourced in `board/lib/types.ts`) — never invent
values:

- **`goal`** — one of `weight_loss`, `sprint_triathlon`, `olympic_triathlon`, `cycling`,
  `swimming`, `running`, `general_fitness`.
- **`level`** — `beginner | intermediate | advanced`.
- **`goalDate`** (`YYYY-MM-DD`, optional) — the race / target date.
- **`daysPerWeek`** (1–7) and **`maxSessionMinutes`** — the availability the plan respects.
- **`sports[]` / `equipment[]`** — from `VALID_ATHLETE_SPORT` / `VALID_ATHLETE_EQUIPMENT`
  (e.g. running, cycling, swimming; bike, pool, gym, treadmill…). What they can/like to do
  and what they have access to.
- **`currentWeightKg` / `targetWeightKg`** (optional) and **`notes`** (freeform).

Gather what's missing **before** sending them to the page — *"to build a good plan I need
your goal, level, how many days a week you can train, and your max session length"*. A
profile save is a single low-stakes write either way.

> **Example.** *"I'm an intermediate runner training for an Olympic triathlon in September,
> I can do 5 days a week, 90 min max, I have a bike, a pool, and a gym."* → tell them to open
> **/athlete** and set: goal `olympic_triathlon`, level `intermediate`, goalDate the Sept
> date, daysPerWeek 5, maxSessionMinutes 90, sports running/cycling/swimming, equipment
> bike/pool/gym. Then offer the training plan (JOB 4).

## JOB 4 — The AI coach: training plan, weekly review, correlations

These are **board AI pages** — each reads the profile + ingested health data and asks
server-side Claude to generate structured output. They are **not** MCP tools, so you **hand
off to the page** (which generates live) and then **interpret** what it produced. Before
sending the user, make sure the **profile is set** (JOB 3) and **recent data is ingested**
(JOB 1) — the coach is only as good as its inputs, and an empty profile 404s the plan.

- **Training plan — `/athlete/training-plan`.** Generates a **personalised week** (one
  entry per day: sport, duration, intensity, effort zones, description) that respects the
  profile's available days / equipment / max session, and **adapts intensity to recovery**
  (HRV, sleep, recent load → a `recovery_status` of good / moderate / poor). Tell the user
  to open the page to generate it; once it's there, help them read it — call out the
  recovery-driven intensity and the rest days, and offer to **push it to the calendar** (JOB
  6). If they have no profile, the page 404s — route them through JOB 3 first.
- **Weekly review — `/athlete/weekly-review`.** A look **back**: training load, sleep, HRV,
  a **form score**, and how the week's nutrition (soft-dep) fit the goal, with advice for
  next week. Use it for *"how was my training this week"*. Interpret the form score + the
  sleep/load balance; keep the not-medical-advice framing.
- **Correlations — `/athlete/correlations`.** Surfaces **sleep ↔ performance** and similar
  links across the logged data (e.g. "your hard sessions the day after <6 h sleep ran
  slower"). Use it for *"how does my sleep affect my training"*. These are **observational
  correlations, not causation** — say so.

> Because these run an LLM on the board, they need `ANTHROPIC_API_KEY` configured in
> `config/secrets.env`; if a page errors with that message, it's a setup gap, not a data
> gap. The coaching prompts already carry a cache_control breakpoint + validate their JSON
> server-side — you don't re-implement any of that here; you supply inputs and interpret.

## JOB 5 — The pre-workout brief ("should I train today?")

**`/athlete/pre-workout-brief`** is the **daily "go / hold back"** read: it folds today's
recovery signals (last night's sleep, morning HRV vs. baseline, resting HR, recent load)
with the profile + the day's planned session into a short brief — *train as planned*, *go
easier*, or *take a recovery day* — with the reasoning. Use it for *"should I train
today?"*, *"am I recovered?"*, *"what's on for today?"*.

Make sure **today's overnight data is ingested** first (JOB 1 — last night's sleep + this
morning's HRV/resting-HR are exactly what the brief leans on); a brief built on stale data
is misleading. Then send the user to the page and **interpret** the recommendation — and
**carry the not-medical-advice framing**: a "go easy / rest" read on low HRV or poor sleep
is a conservative training default, and any pain, illness, or abnormal symptom means
**stop and consult a professional**, not push through.

## JOB 6 — Push a plan to the calendar; ingest a report to the vault

**A. Push the generated plan to the calendar.** Once a training plan exists on
`/athlete/training-plan`, the user can **materialise it as calendar events** (one event per
training day, with the session in the description) via the page's "add to calendar" action
(`POST /api/athlete/push-plan-to-calendar` under the hood). This is a **gated write** (404s
if the add-on is off) and a **bulk** one (a whole week of events) — so in approval mode,
confirm before it fires. The events land in the board's calendar (visible to the `calendar`
MCP / the calendar UI). Rest days are written as "Rest" markers. Don't push a plan to the
calendar unless asked.

**B. Ingest a health report to the vault.** `ingest_health_to_vault({days})` **composes** a
Markdown health summary of the last N days (default 7) and returns it as
`{ vault_ingest_content, domain: "life", instruction }` — it does **NOT** write to the vault
itself. To actually persist it, take the returned `vault_ingest_content` and pass it as the
`content` argument (with `domain: "life"`) to the **`vault`** MCP's `ingest` tool (which is
async — submit, then poll `ingest_status`; see `/vault-operations`). Use this for *"save my
health summary to my knowledge base"*, *"log this week's training to the vault"*. The
compose step is an ungated read; the vault write is governed by the vault add-on.

## Conventions (guardrails recap)

- **`health` MCP only, via the tools, for health DATA.** Never `bash`/`curl`. The board UI
  (`/health` + `/athlete/*`) is the read twin + the AI-generation surface; you do the
  ingestion + the reads + the orchestration.
- **The add-on must be ENABLED for writes.** A disabled add-on 404s every write ("Not found
  — the health add-on may be disabled.") while reads stay open — tell the user to flip it on
  at **/addons**; you don't enable it yourself. A **401** (not 404) is a token problem
  (`HEALTH_PUSH_TOKEN`), a setup issue for `/health-mcp-setup`.
- **Mode (STEP 0):** auto → just do it; approval → confirm **bulk** writes (a big backfill,
  a whole week pushed to the calendar) and **destructive** ones (`delete_health_data`, a
  wholesale profile replace) **in chat** before firing. A single write is low-stakes either
  way. **There is no pending/propose queue** — confirmation is conversational.
- **Canonical taxonomy, always.** Seven types (`workout`, `sleep_night`, `sleep_nap`,
  `hrv`, `resting_hr`, `steps`, `vo2max`); the metric goes in **`data.value`** (sleep =
  hours, with stages in `data.metadata`); workouts carry `activity` + `duration_min` on
  `data`. `ts` = `YYYY-MM-DD` for daily/sleep, full ISO for a workout. Stable unique `id`
  per logical entry so a re-push overwrites. Never use `data.bpm`/`data.count`/`data.ms`.
- **Reads:** `get_daily_summary` (the day, incl. nutrition + calorie balance),
  `get_health_summary` (per-type aggregates over a range), `get_health_trends` (the series),
  `list_health_data` (raw rows). Read the contract field names, not the producer's shape.
- **Profile + AI coaching live on the board pages.** The profile (`/athlete`), training plan,
  weekly review, pre-workout brief, and correlations are **board AI surfaces, not MCP
  tools** — gather the inputs, send the user to the page to generate, then **interpret** the
  output. Bind the profile to the **English `VALID_ATHLETE_*` enums**; gather missing fields
  conversationally; require a profile + recent data before generating.
- **Calendar + vault.** A generated plan → the calendar via the page's push action (gated,
  bulk — confirm in approval mode). A health report → the vault via
  `ingest_health_to_vault` (compose) **then** the `vault` MCP's `ingest` (the actual,
  async, write).
- **Soft dependency on Nutrition.** The food + calorie-balance parts of `get_daily_summary`
  / the weekly review read the Nutrition add-on; if it's off, those sections are just empty —
  not a fault. Point the user at `/nutrition-chef` if they want them.
- **Deletes are HARD.** `delete_health_data` has no soft-archive — it's irreversible, unlike
  the board's soft `archive_case`. Confirm before deleting in approval mode.
- **NOT MEDICAL ADVICE.** Plans, recovery reads, reviews, and briefs are informational
  estimates — **say so**, and **defer injuries, medical conditions, abnormal symptoms,
  pregnancy, or an under-18 user to a physician / physiotherapist / qualified coach**
  (recommend they consult one; don't push hard training).
- **Report** what you wrote / read: the accepted/purged counts on a push, the day's summary
  + calorie balance, the trend direction, the recovery status driving a plan, and the ids /
  event count when you push to the calendar.
